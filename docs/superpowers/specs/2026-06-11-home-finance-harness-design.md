# 주거금융계산기 MVP 설계

## 1. 목적

`주거금융계산기`는 한국의 전세, 임대차, 전세대출 정보를 다루는 정적 우선 금융 정보 사이트다. 사용자는 브라우저에서 보증금 인상액과 대출 이자를 계산하고, 공식 출처를 근거로 작성된 가이드를 읽을 수 있다.

이 프로젝트의 핵심은 페이지 수가 아니라 안전한 운영 하네스다. Codex는 공식 출처를 조사해 콘텐츠와 코드를 제안하지만, 자동으로 운영 배포하지 않는다. 모든 변경은 Git 브랜치와 PR, Vercel Preview, 사람의 검토를 거쳐 병합한다.

## 2. MVP 범위

### 포함

- Next.js App Router 기반 반응형 웹사이트
- 계산기 3개
  - 전세보증금 5% 인상 계산기
  - 전세대출 월 이자 계산기
  - 전세대출 증액 계산기
- 실제 게시 가능한 공식 출처 기반 가이드 3개
  - HUG, HF, SGI 보증 차이
  - 전세대출 연장과 대환대출 차이
  - 계약갱신요구권 사용 시 주의사항
- 사이트 소개, 문의, 개인정보처리방침, 이용약관
- 공통 금융 면책 문구와 출처 표시
- AdSense 승인 준비용 광고 슬롯 컴포넌트
- Codex 작업 규칙, 반복 작업 Skills, 콘텐츠 검증 명령
- 단위 테스트, 린트, 타입 검사, 프로덕션 빌드 CI
- Vercel 배포 설정과 환경변수 예시

### 제외

- 금융상품 추천, 중개, 상담 신청 수집
- 대출 승인 가능성 또는 개인별 한도 판정
- 회원 가입, 데이터베이스, 관리자 CMS
- 계산기 입력값 저장 또는 서버 전송
- 은행 금리 자동 수집과 자동 게시
- 운영 환경으로의 무검토 자동 배포

## 3. 기술 구조

- Next.js App Router와 TypeScript
- Tailwind CSS
- MDX 기반 가이드 콘텐츠
- Vitest와 Testing Library
- ESLint 및 TypeScript 검사
- Vercel Preview와 Production 배포

계산 로직은 `src/calculators`에 순수 함수로 둔다. 클라이언트 컴포넌트는 입력 상태와 표시만 담당한다. 가이드 콘텐츠는 `content/guides`의 MDX 파일로 관리하고, frontmatter와 출처 ID를 빌드 전에 검증한다.

예상 구조:


## 4. 화면 설계

시각 방향은 **B안, 실용 계산기형**으로 확정한다. 첫 화면에서 방문 목적을 바로 해결하고, 결과와 관련된 공식 가이드로 자연스럽게 이동시킨다.

### 홈

- 헤더: 사이트명, 계산기, 금융 가이드, 소개
- 히어로: "전세보증금과 대출 이자를 바로 계산하세요"
- 개인정보 안내: 입력값은 저장하거나 전송하지 않음
- 핵심 계산기 카드 3개
- 공식 출처 기반 가이드 카드 3개
- 최근 콘텐츠 확인일과 정보 제공 원칙
- 공통 면책, 정책, 문의 링크가 있는 푸터

광고는 핵심 입력 폼과 계산 결과 사이에 배치하지 않는다. 환경변수에 광고 ID와 슬롯 ID가 없으면 광고 마크업과 빈 공간을 모두 렌더링하지 않는다.

### 계산기 공통 UX

- 원 단위 숫자 입력과 한국식 금액 보조 표기
- 입력 즉시 또는 명시적 계산 버튼으로 결과 산출
- 빈 값, 음수, 허용 범위 초과 값을 필드 가까이에서 안내
- 결과를 `예상 계산값`으로 명시
- 실제 금리, 한도, 승인 여부는 금융기관 심사에 따라 달라짐을 안내
- 관련 가이드와 공식 출처 링크 제공

## 5. 계산 규칙

### 전세보증금 인상

- 입력: 기존 보증금, 신규 보증금
- 출력: 인상액, 인상률, 5% 초과 여부
- 인상률: `(신규 보증금 - 기존 보증금) / 기존 보증금 * 100`
- 기존 보증금이 0 이하이면 계산하지 않는다.
- 5% 표시는 법률 판단이 아니라 입력값의 산술 비교 결과임을 명시한다.

### 월 이자

- 입력: 대출 원금, 연 금리
- 월 이자: `대출 원금 * 연 금리 / 100 / 12`
- 연 이자: `대출 원금 * 연 금리 / 100`
- 원금 균등, 원리금 균등 상환액이 아니라 이자만 계산한다.

### 전세대출 증액

- 입력: 기존 대출금, 기존 연 금리, 증액 금액, 증액분 연 금리
- 기존 월 이자와 증액분 월 이자를 각각 계산
- 증액 후 월 이자는 두 값을 합산
- 월 이자 증가액은 증액분 월 이자와 동일
- 실제 금융기관이 전체 잔액에 신규 금리를 적용할 수 있음을 별도 안내한다.

모든 금액 계산은 부동소수점 표시 오차가 사용자에게 노출되지 않도록 원 단위 반올림 규칙을 명시하고 테스트한다.

## 6. 콘텐츠 모델과 공식 출처

각 MDX 가이드는 다음 frontmatter를 필수로 가진다.

```yaml
title: string
description: string
slug: string
updatedAt: YYYY-MM-DD
reviewStatus: reviewed
sources:
  - registered-source-id
disclaimer: financial-general
```

공식 출처는 `src/data/sources.ts`에 ID, 기관명, 문서명, URL, 확인일을 구조화해 등록한다. 초기 조사 대상은 다음 공식 기관으로 제한한다.

- 금융위원회
- 금융감독원
- 은행연합회 소비자포털
- 한국주택금융공사(HF)
- 주택도시보증공사(HUG)
- SGI서울보증
- 국가법령정보센터
- 필요한 경우 개별 은행 공식 상품 설명서와 공지

검색 결과나 언론 보도는 공식 원문을 찾기 위한 단서로만 사용한다. 본문의 조건, 수치, 법률 설명은 공식 원문과 대조한다. 구현 시점의 최신 공식 페이지를 다시 열어 URL, 시행일, 문구를 확인하며, 자동화가 접근하지 못하는 페이지는 사람의 확인 대상으로 표시한다.

## 7. 콘텐츠 검증

`npm run validate:content`는 다음 조건을 검사한다.

- 필수 frontmatter 존재 및 형식
- 출처 ID가 레지스트리에 등록되어 있음
- 최소 하나 이상의 공식 출처 존재
- 업데이트일과 공통 면책 식별자 존재
- `reviewStatus`가 `reviewed`인 문서만 공개 목록에 포함
- 확정적 또는 과장된 금융 표현 금지
- 내부 링크 대상과 필수 정책 페이지 존재
- canonical slug 중복 금지

초기 금지 표현에는 다음을 포함한다.

- 무조건 가능
- 반드시 승인
- 누구나 가능
- 100% 보장
- 최저금리 확정
- 가장 좋은 대출

검증기는 단어 존재만으로 오탐이 날 수 있으므로, 금지 표현을 설명하는 정책 문서와 테스트 fixture는 공개 콘텐츠 검사 대상에서 제외한다.

## 8. 오류와 개인정보 처리

- 계산 입력값은 브라우저 메모리에서만 처리한다.
- 서버 액션, API, 로그, 쿠키, 로컬 스토리지로 입력값을 보내거나 저장하지 않는다.
- 분석 도구를 추가하더라도 계산 실행 횟수나 입력값을 수집하지 않는다.
- 계산 오류는 사용자에게 수정 가능한 입력 안내로 표시한다.
- 출처 접근 실패는 기존 콘텐츠를 자동 변경하지 않고 검토 실패로 보고한다.
- 법령과 상품 조건의 최신성을 보장하지 않으며 업데이트일과 공식기관 재확인 안내를 표시한다.

문의 페이지는 MVP에서 이메일 주소 또는 외부 문의 링크만 제공하며, 사용자 금융정보를 받는 자체 폼은 만들지 않는다.

## 9. Codex 운영 하네스

### AGENTS.md

프로젝트 목적, 공식 출처 우선순위, 개인정보 금지, 계산기 구현 규칙, 콘텐츠 구조, 완료 조건, PR 작성 형식을 정의한다. Codex는 계산식이나 금융 설명을 변경할 때 출처와 테스트 영향을 함께 보고한다.

### Skills

- `content-writer`: 공식 출처 조사 후 MDX 초안 작성
- `fact-checker`: 출처, 날짜, 위험 표현, 면책, 계산 예시 검토
- `calculator-builder`: 순수 함수, 입력 경계, 단위 테스트 작성
- `seo-reviewer`: 검색 의도, 메타데이터, FAQ, 중복 콘텐츠 검토

각 Skill은 한 가지 반복 작업에 집중하고 출력 형식을 명시한다. 프로젝트 전역 안전 규칙은 중복하지 않고 AGENTS.md를 기준으로 삼는다.

## 10. 배포와 AdSense 준비

- PR마다 Vercel Preview 생성
- 사람이 Preview와 출처를 검토한 후 병합
- `main` 병합만 Production 배포 대상으로 설정
- `NEXT_PUBLIC_ADSENSE_CLIENT_ID`와 슬롯 ID는 환경변수로 관리
- 값이 없으면 광고 스크립트를 로드하지 않음
- 광고 슬롯에는 `광고` 레이블을 표시
- 개인정보처리방침에 실제 사용 중인 쿠키와 광고 기술만 명시

AdSense 승인을 보장하는 표현은 사용하지 않는다. 이 MVP는 정책 페이지, 독자적 콘텐츠, 명확한 탐색 구조와 광고 배치 경계를 제공하는 승인 준비형 구조다.

## 11. 테스트와 완료 조건

### 자동 검증

- 계산 함수의 정상값, 0, 음수, 소수 금리, 큰 금액, 5% 경계 테스트
- 계산기 입력 오류와 결과 렌더링 테스트
- 콘텐츠 frontmatter, 출처, 금지 표현 검증 테스트
- 주요 페이지 렌더링과 내부 링크 확인
- `npm test`
- `npm run lint`
- `npm run typecheck`
- `npm run validate:content`
- `npm run build`

### 수동 검증

- 모바일과 데스크톱 레이아웃
- 키보드 입력과 포커스 표시
- 계산 결과의 단위와 반올림
- 각 가이드의 공식 출처 원문과 확인일
- 광고 환경변수 미설정 상태에서 빈 광고 영역이 없는지 확인
- Vercel Preview에서 정책 페이지와 canonical URL 확인

## 12. 성공 기준

- 세 계산기가 정의된 산식과 경계 규칙대로 동작한다.
- 세 가이드가 구현 당일 확인한 공식 출처, 업데이트일, 면책 문구를 갖는다.
- 계산 입력값이 네트워크나 영구 저장소로 전달되지 않는다.
- 모든 자동 검증과 프로덕션 빌드가 통과한다.
- B안의 계산기 중심 홈에서 각 계산기까지 한 번의 선택으로 이동할 수 있다.
- Codex가 동일한 규칙으로 새 계산기와 가이드를 제안할 수 있는 운영 문서와 Skills가 존재한다.
